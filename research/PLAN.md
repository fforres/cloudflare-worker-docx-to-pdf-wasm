# wasm-docx-to-pdf — master plan

## Goal
A Rust library that compiles to `wasm32-unknown-unknown`, deployable inside a Cloudflare Worker (10 MiB compressed bundle ceiling, 128 MB isolate memory, 5 min CPU on paid plan), converting DOCX to PDF while preserving **content structure** (text, headings, lists, tables, images). Exact font/pagination fidelity is **not** a goal.

## Deployment shape (hybrid)
```
HTTP POST /convert (DOCX body)
        │
        ▼
   JS Worker (Cloudflare)
        │  ── instantiates WASM at module top-level
        ▼
   wasm-docx-to-pdf.wasm  ── Rust core
        │
        ▼
   PDF bytes returned to client
```

## Folder map
- `fixtures/{tier1,tier2,tier3}/` — DOCX inputs.
- `reference-pdfs/{tier1,tier2,tier3}/` — ground-truth PDFs generated by LibreOffice headless from those DOCX files.
- `harness/` — comparison tool (text similarity, page/image counts, scorecard writer).
- `approach-a-custom/` — `docx-rs` + `cosmic-text`/`taffy` + `krilla`. Own `PLAN.md`.
- `approach-b-typst/` — DOCX → Typst markup → `typst-pdf`. Own `PLAN.md`.
- `approach-c-rdocx/` — `rdocx` / `rdocx-wasm`. **Preferred path.** Own `PLAN.md`.
- `results/` — per-approach scorecards and the final comparison.

## Phases
1. **Phase 0 — Setup**: tooling, folder structure, fixtures, reference PDFs. *(this PR)*
2. **Phase 1 — Harness**: a comparison tool that scores generated-PDF vs reference-PDF.
3. **Phase 2 — Three PoCs in parallel**. Each must:
   - Build natively (CLI binary that takes `input.docx` → `output.pdf`).
   - Build to `wasm32-unknown-unknown`.
   - Report compressed `.wasm` size after `wasm-opt -Oz`.
   - Run the full test corpus and produce its PDFs.
4. **Phase 3 — Score and pick a winner**.
5. **Phase 4 — Minimal CF Worker** wrapping the winner.

## Hard constraints (decision criteria)
| Constraint | Limit | Action if exceeded |
|---|---|---|
| Compressed WASM | ≤ 10 MiB | Disqualify approach, or move to Containers (out of scope for PoC). |
| Isolate memory | ≤ 128 MB | Profile peak heap; if it exceeds 100 MB on a 50-page doc, flag as risky. |
| CPU per request | ≤ 300 s (paid max) | If conversion exceeds 5 s on a 20-page doc, flag as risky. |

Startup/instantiation cost is **explicitly out of scope** per project owner — top-level `WebAssembly.instantiate` amortizes it.

## Fidelity rubric (per `TEST_PLAN.md`)
- **Critical**: text token recall vs reference ≥ 0.85.
- **Important**: page count within ±20 % of reference; image presence/count match; table count match.
- **Nice-to-have**: SSIM ≥ 0.5 on first-page render; heading hierarchy preserved.
- **Out of scope**: font identity, exact line breaks, kerning, hyphenation.

## What we are explicitly NOT building
- A LibreOffice/Word replacement.
- Anything that handles SmartArt, charts, equations (OMML), or track-changes rendering in v1.
- Font fidelity (Calibri → Calibri). We bundle one or two OFL fonts and accept reflow.
- Image format conversion beyond what the chosen PDF library does natively.

## Single biggest unknown (still)
Compressed WASM size for B and C. Approach C (`rdocx-wasm`) is the project owner's preferred path, so we land it first.
