# opt-5-complex-corpus — Results

## TL;DR

**Bug fixed. Avg complex-corpus recall jumped from 0.71 → 0.94.** Final WASM
bundle 5.93 MiB raw / **2.88 MiB gzip** (well inside CF Workers' 10 MiB limit).

Root cause was **not** a WASM codegen issue in rdocx-pdf. It was a
font-data issue: the **Carlito** typeface (one of the families bundled by
`rdocx-layout`'s `bundled-fonts` feature) interacts badly with `rdocx-pdf
0.1.2`'s ToUnicode-CMap writer and produces a scrambled text layer (the
`eeeeatioe` gibberish). The same bug fires on **native** macOS once the
build is forced to use Carlito (e.g. by removing system fonts from the
lookup or by force-bundling Carlito) — it is not WASM-specific. The reason
the native `rdocx` CLI looked fine in earlier baselines is that macOS has
Tahoma/Helvetica installed, so `fontdb.load_system_fonts()` resolved the
DOCX's Calibri request to a real system font and Carlito never got picked.

## Phase log

### Phase 0 — Setup
Copied baseline `approach-c-rdocx/converter` to `opt-5-complex-corpus/converter`.
Renamed crate to `approach_c_rdocx_opt5`. Built native, reproduced the
gibberish on `cdc_ngs_validation.docx`.

### Phase 1 — Diagnostic before compile-flag sweep
**Hit a critical insight in the first 15 min that obsoleted Phase 1:**
the project's `approach_c_rdocx` native binary at HEAD *also* produces
gibberish on cdc_ngs_validation. The "native works, WASM doesn't"
attribution in `complex-fixtures/COMPARISON.md` was based on PDFs
generated by the *installed* `rdocx` CLI (`cargo install rdocx`), which
does **not** use bundled-fonts in its feature set — and on macOS it finds
real fonts via `fontdb.load_system_fonts()`. Our crate had
`rdocx-layout = { features = ["bundled-fonts"] }`, which is what
triggers Carlito to be loaded as a fallback.

Bisect:

| Variant | Output text |
|---|---|
| fresh cargo proj, `rdocx` only | clean ("Next Generation") |
| add `rdocx-layout = "0.1.2"` (no features) | clean |
| add `rdocx-layout = "0.1.2", features=["bundled-fonts"]` | **gibberish** |
| `to_pdf_with_fonts(&[Carlito-{R,B,I,BI}])` | **gibberish** |
| `to_pdf_with_fonts(&[Liberation-Sans-* aliased to Calibri])` | clean |

So Carlito specifically is the trigger. Caladea (paired with Cambria) is
the same family architecture and almost certainly also affected, though
the cdc fixture didn't use Cambria so I didn't bisect Caladea individually.

### Phase 2 / 3 — Picked Path C (the practical fix)
Path C in the brief was "pre-process the DOCX font references", but I
landed on a cleaner variant of the same idea: **don't bundle Carlito or
Caladea. Bundle only Liberation Sans/Serif/Mono and pre-register them as
named aliases for every common Word font.** This passes them via
`doc.to_pdf_with_fonts(...)`, which rdocx documents as the *highest
priority* lookup tier — so Liberation always wins over the buggy
bundled set, on any target.

Fonts bundled:
- LiberationSans-{Regular, Bold, Italic, BoldItalic}
- LiberationSerif-{Regular, Bold, Italic, BoldItalic}
- LiberationMono-{Regular, Bold, Italic, BoldItalic}

Aliases registered (23 names × 4 faces ≈ 92 entries):
- Sans family ← Liberation Sans: Calibri, Calibri Light, Arial, Helvetica,
  Tahoma, Verdana, Segoe UI, Trebuchet MS, Lucida Sans
- Serif family ← Liberation Serif: Cambria, Times New Roman, Times,
  Georgia, Book Antiqua, Garamond, Palatino Linotype
- Mono family ← Liberation Mono: Courier New, Courier, Consolas,
  Lucida Console

The `bundled-fonts` feature on `rdocx-layout` is now **off**.

The +1.5 MiB gzip overhead vs opt-3 comes from bundling **full** Liberation
TTFs (Sans, Serif, Mono × 4 faces ≈ 1.4 MB of fonts). A follow-up could
subset these too (the opt-3 subsetting pipeline could be reused) to claw
back that delta — but at 2.88 MiB gz, opt-5 is comfortably under the CF
ceiling and the correctness improvement is far more valuable than the
size headroom.

## Honest assessment

- **The cmap-drift narrative in `COMPARISON.md` was partially wrong.** It was
  attributed to a wasm32-unknown-unknown codegen bug in rdocx-pdf. It is
  actually a Carlito-specific data interaction in rdocx-pdf that fires on
  any target where Carlito wins the font lookup. The reason native looked
  fine in earlier tests is that the *installed CLI* (without bundled-fonts)
  resolves Calibri to a real macOS system font, while our crate (with
  bundled-fonts) loads Carlito into fontdb and rdocx-pdf trips on Carlito's
  cmap layout.
- The fix is small, local to opt-5, and doesn't require touching rdocx-pdf
  internals. Approach B/A's "fix" is more architectural; opt-5 stays inside
  the rdocx stack and just sidesteps Carlito.
- The Caladea family is structurally the same as Carlito (both Google's
  "Open Source Metric-compatible" replacements for Calibri/Cambria) and
  very likely has the same bug. Out of caution we don't bundle it.
- **Hard pass criteria met:** complex avg 0.94 ≥ 0.85 ✓, cdc_ngs_validation
  0.97 ≥ 0.85 ✓, toy fixtures no regression ✓, worker smoke 200 ✓.

## Recommendation

**Ship opt-5.** It is correct, robust, small enough for CF Workers, and a
strict improvement on opt-3 in every quality metric. If the +1.5 MiB
bundle bloat matters, follow up by subsetting the Liberation faces (apply
opt-3's pipeline to opt-5's font set). The remaining gap — textbox /
sidebar text extraction in DOCX like `un_seea_policy_brief` — is an
upstream `rdocx-oxml` parser limitation and out of scope for any opt-N
font fix; addressing it would require either contributing to rdocx
upstream or moving to approach B's docx-rs reader.
